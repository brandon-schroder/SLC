# Guide trace scripts — the source of truth for the guides' numbers

These scripts regenerate the *measured* numbers cited in the `docs/guide/`
series. They exist because this solver is under active development (loss
correlations are being calibrated, drivers tuned), so the **performance
numbers in the guides are a moving target** — while the *mechanics* the
guides explain are stable.

## The convention

- **Structural facts** (iteration structure, the Euler identity, typed
  statuses, the stabilization mechanism, sign conventions, the coupling
  structure) go in the guide **prose**. They do not move.
- **Calibration-dependent magnitudes** (PR, η, entropy spreads, exact
  iteration counts) go in a clearly-marked **snapshot block** in the guide,
  stamped with the commit they were measured at — and their generator is one
  of these scripts. A stale snapshot is then a one-command refresh, not a
  manual re-derivation.

When a guide's numbers drift (a closure gets recalibrated, a driver constant
changes), the fix is:

1. `<project-venv>/python docs/guide/_traces/trace_guideN.py`
2. Paste the labeled block into that guide's snapshot table.
3. Bump the guide's commit stamp to the current `git rev-parse --short HEAD`.

No archaeology. If a number appears in guide *prose* and keeps drifting,
migrate it into the snapshot block — prose should not carry volatile values.

## The scripts

| Script | Feeds | Notes |
|---|---|---|
| `trace_guide1.py` | Guide 1 run table, convergence tables, Euler check, run C, choke example | fast (~1 min) |
| `trace_guide2.py` | Guide 2 §8 worked conversions + closing check | fast; the B.2/B.3/B.4 conversions are fixed illustrations and do **not** move |
| `trace_guide3.py` | Guide 3 §4.4 refutation table, §4.5 Newton record, §5 mixing | slow (~2–3 min; V7/V8 Tier-3 runs) |

The Wilkinson envelope (Guide 3 §2.1, Appendix C.3) is generated by
`tools/calibrate_wilkinson.py`, the canonical tool — not duplicated here.

## Health signals to watch when refreshing

- **`validity`** on a converged run. `validity → 0` means the closures are
  fully out of calibration for that case's operating point — the solve
  converged but the *performance* numbers are running in a saturated regime
  and should not be read as clean measurements. (As of this writing the
  `V5MultistageCompressor` showcase sits at validity 0 after the Lieblein
  ω̄-inversion fix; its metal angles want re-tuning to the corrected loss.)
- A **status** that is not `CONVERGED` — the guide narrative around that
  number may need revisiting, not just the digits.

These are documentation-support scripts, not the test suite; they are not
run by CI. The permanent regressions live in `tests/`.
